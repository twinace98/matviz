## Autonomous judgments recorded

### J1. rAF 30fps target via `performance.now()` throttle

`requestAnimationFrame`은 보통 60fps 이상 호출. 매 callback에서 frame 증가하면
60fps 재생 (사용자가 너무 빠르다고 느낄 수 있음 + 1k atom×1k frame perf 부담).

**선택**: rAF 콜백마다 `performance.now()` 비교, `1000/30 ≈ 33.3ms` 경과시에만
frame 증가. 33ms 미만이면 그냥 다음 rAF 스케줄. 결과: 30fps cap.

대안 (`setInterval(33)` 사용)은 tab inactive 시 정확도 저하 + rAF와 sync
안됨. rAF + throttle이 표준 패턴.

### J2. Loop wrap (마지막 frame → 첫 frame)

`(currentFrame + 1) % frameCount`로 자동 wrap. 사용자가 explicit pause 안 하면
무한 재생. 일반 MD trajectory 시각화 패턴.

### J3. User scrub auto-pause

slider input 이벤트는 사용자 드래그를 의미. 재생 중에 slider 만지면 의도가
"이 frame으로 가서 멈추기"라고 해석 → trajPlaying = false 자동 전환. UX
pattern 통상.

### J4. loadStructure case에서도 visibility 갱신

WebView가 sticky — 같은 webview 인스턴스에서 multi-frame trajectory 열고 그
다음 single-frame 파일 열면 trajectory section이 잔재. `loadStructure`에서
`updateTrajectorySectionVisibility()` 호출 → renderer.hasTrajectory()가 false
반환 → section hide.

대칭으로 trajectory 재생 중 single-frame 로드 시 `trajSetPlaying(false)` 안에서
rAF cancel (visibility 갱신 시 hasTrajectory()=false → 자동으로 trajPlaying이
false 되도록 처리).

### J5. 자동 검증 한계 — webview UI

UI 자체 자동 검증은 visual harness 외 (CLI Phong path)에서 불가능. tsc + build
+ visual 회귀 0이 commit 전 게이트. 실제 재생 동작은 user-side VSCode install
+ trajectory fixture 열기 검증.

### J6. 30fps target 도달 가능성

1k atom 재생에서 30fps 달성 여부는 17.1.5 (bond toggle off + cachedBonds
inherit) 후 측정. 현 17.1.4는 default off (bond 매 frame 재계산 안 함, 17.1.5
도입 전이라 cached bond 재사용 — 다행히 renderer.setFrame 자체가 cachedBonds
clear 안 함). N=2 atoms × 4 frames AXSF 기준 trivially 30fps. 큰 fixture는
17.1.5 perf 측정.

## Known limitations / deferred

1. **Frame perf in 1k+ atoms** — 17.1.5에서 bond recompute toggle + cached
   bond reuse로 큰 N에서도 30fps 달성.
2. **Keyboard shortcut (Space for play/pause)** — UI button만 첫 컷.
3. **재생 속도 슬라이더** — 30fps 고정. 사용자 "fast/slow" 토글은 v0.17.x.
4. **Frame counter 키보드 입력** — slider 드래그/키보드 화살표만. 직접 frame
   번호 입력은 v0.17.x.
5. **Loop on/off 토글** — 자동 wrap만. once-only는 v0.17.x.
