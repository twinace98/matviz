## Autonomous judgments recorded

### J1. Single secondary phase가 아닌 array

Plan은 "two structures rendered simultaneously"이지만 array 기반으로 구현 →
3개 이상도 가능. addPhase는 push, clearPhases는 wipe-all. 단순한 add/clear
시맨틱 → first cut에 적합.

대안 (single-slot, addPhase가 기존 phase 대체)은 단순하지만 "compare 3 phases"
워크플로우 차단.

### J2. Default offset (0, 0, 0)

Plan §J9 "offset 명시 또는 cell-overlap" 중 cell-overlap을 default로 — 두
phase가 같은 좌표 공간에 겹쳐 보임. 시각적 차이는 transparency로만 (default
opacity 0.5). 사용자 워크플로우: 두 phase가 같은 결정 시스템의 변형 비교
(예: relaxed vs unrelaxed)일 때 overlap이 가장 직관적.

명시 offset은 v0.17.x에서 사이드 패널 슬라이더로 추가 가능.

### J3. Bonds/boundary 부재 (atom only)

Secondary phase는 atom만 렌더, bond/boundary 미렌더. 이유:
- Visual clutter 회피 (primary와 secondary bond 겹치면 혼란).
- Perf: 각 phase마다 detectBonds 다시 돌리는 건 비싼 작업.
- Comparison/overlay 핵심 정보는 atom 위치 — bond는 시각적 보조.

향후 사이드 패널 phase 관리 UI에서 per-phase bond 토글 가능 (v0.17.x).

### J4. Lazy import in extension.ts

`parseStructureFile`을 top-level import 대신 dynamic `await import('./parsers/index.js')`.
이유: extension.ts는 이미 crystalEditorProvider를 통해 parsers 의존, top-level
double import 시 일부 빌드 환경에서 circular module load 경고 가능. lazy로
회피. perf 영향 무시 가능 (사용자 명시 명령 시점).

`.js` 확장자 명시 — esbuild의 ESM resolution 호환.

### J5. transparent material + depthWrite=false + renderOrder=1

v0.16.2 partial occupancy 패턴 carry-over. Stacked transparent atoms blend
correctly over opaque primary.

### J6. Side-panel phase 관리 UI 미구현

Plan §17.2은 "사이드 패널에 phases 목록 + visibility/opacity 토글" 명시.
첫 컷에서는 명령 팔레트만 (add + clear). 이유:
- Phase 관리 UI는 list rendering + per-phase controls + 추가 메시지 타입 (
  `setPhaseOpacity`, `togglePhase` 등) 필요 → 큰 작업.
- 명령 팔레트 add/clear로 워크플로우 가능 (compare 두 결정 → addPhase →
  관찰 → clearPhases → 다른 두 결정).

v0.17.x에서 사이드 패널 추가 권장.

### J7. Trajectory phase 미지원 첫 컷

Plan §J9 "두 번째 phase도 trajectory 가능". 첫 컷에서는 single-frame
CrystalStructure만 입력 받음 (parseStructureFile 사용). trajectory 입력은
parseStructureFileTraj + 첫 frame만 사용 또는 trajectory 자체를 secondary
phase로 등록 (각 phase별 frame index 유지) → 복잡도 증가. v0.17.x.
