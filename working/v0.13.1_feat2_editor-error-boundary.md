# v0.13.1 · Feature 2 — Editor error boundary

**Status:** complete
**Date:** 2026-04-17

## Goal

When `parseStructureFile` throws inside `openCustomDocument`, VSCode previously showed a generic "Failed to open" error with no recourse. Surface the real parser message and offer a one-click path to open the file as plain text.

## Change

`src/editor/crystalEditorProvider.ts:44-61` — wrapped the parse call in try/catch. On failure: `showErrorMessage(...)` with an "Open as Text" action that runs `vscode.openWith` against the default editor, then rethrows so VSCode still marks the open as failed.

## Validation

- `npx tsc --noEmit` → no errors
- `npm run build` → Build complete

Manual toast behavior will be verified during the 13.1.3 round once the `priority: "option"` + title-bar button work lands, since both affect the "unparseable file" flow.
