## Autonomous judgments recorded

### J1. Plan-assumption violation: CLI has no impostor path

**Discovery:** Grepping `scripts/render.ts` for "impostor"/`ShaderMaterial`/`linearToSRGB` returned 0 matches. CLI renderer's atom rendering (line 451–457) uses `THREE.MeshPhongMaterial` + `THREE.SphereGeometry` + `THREE.InstancedMesh` only. No impostor shaders are inlined.

This contradicts the plan's working assumption that the harness would invoke CLI twice with `--impostor on/off` and diff the two outputs.

**Three options considered:**
- **Option A: Port impostor shaders to CLI inline.** ~300 LOC port (sphere ~100 + cylinder ~200), creates permanent two-source maintenance debt (every shader change must mirror in webview AND CLI). Rejected.
- **Option B: Build webview-stub Puppeteer harness.** Load `dist/webview.js` IIFE in Puppeteer with `acquireVsCodeApi` stub, drive via postMessage. Blocked by IIFE encapsulation — webview internals (renderer, `setImpostorEnabled`) are not exposed for programmatic control. Custom diagnostic-page bundle (~1 hour) needed. Tried mentally; estimated > 30 min budget. Deferred.
- **Option C: Drop impostor parity gate; ship CLI regression harness alone.** **Chosen.** Regression harness still serves its primary value (gating 16.1+ rendering changes). Impostor fix becomes its own deliverable.

**Why Option C:**
1. The harness's value to v0.16 is *regression catching for 16.1~16.5*, all of which use the Phong path (ellipsoid sites are explicitly Phong per plan §결정 게이트 #1; magnetic arrows are new InstancedMesh; partial occupancy reuses sphere path). CLI Phong-only coverage is sufficient.
2. The impostor fix is a v0.15 known-issue carry-over, isolated from 16.1+ work. Splitting doesn't block downstream features.
3. Diagnosis (§J3 below) is contradictory; without measurement infrastructure the fix would be a guess. Better to defer than ship a guess.

### J3. Impostor diagnosis (analytical, no measurement)

Read `sphereImpostor.ts` + `cylinderImpostor.ts` + scene lighting in `renderer.ts`:

| 후보 원인 | 결과 |
|----------|------|
| 광원 미스매치 | ❌ — webview scene + impostor uniforms 모두 ambient 0.5 + dir 0.8 + dir 0.3 매칭 |
| sRGB 인코딩 | ❌ — impostor의 `linearToSRGB`는 Three의 `<colorspace_fragment>` chunk와 동일 공식 (1.055/2.4 cutoff at 0.0031308). outputColorSpace=sRGBColorSpace 기본값 (Three 0.170, post-r152). |
| BRDF_Lambert 정규화 | ❌ — v0.15.0에서 이미 fix됨 (sphereImpostor.ts:96–101 주석). ambient + directional 둘 다 RECIPROCAL_PI 통과. |
| 톤맵 | ❌ — `toneMapping` 명시 안 함, default `NoToneMapping` (no-op). |
| **Shininess + specular factor** | ⚠️ 미스매치. Phong=30 (renderer.ts:1945), impostor sphere=80, impostor cylinder=40. Impostor specular 계수도 empirical 0.067. |

**Bulk diffuse 수학 비교** (all atoms facing both lights at peak, white element color):

```
Three Phong:     (ambient + dir1*ndotl + dir2*ndotlFill) * RECIPROCAL_PI * material.color
                = (0.5 + 0.8 + 0.3) * 0.318 * color = 0.508 * color  ✓
Impostor:       vColor * (uAmbient + ndotl*0.8 + ndotlFill*0.3) * RECIPROCAL_PI
                = vColor * (0.5 + 0.8 + 0.3) * 0.318 = 0.508 * vColor  ✓ 매칭
```

**Specular peak 비교** (dotNL=dotNH=dotRV=1):

```
Three BlinnPhong: irradiance * (RECIPROCAL_PI * (shininess/2 + 1)) * specularColor
                 = (1 * 0.8) * (0.318 * 16) * 0.0666 = 0.271  per channel
Impostor Phong:  pow(dotRV, 80) * 0.067 = 0.067  per channel
```

**Math가 맞다면 impostor의 specular는 Phong보다 ~4× 어두워야 함.** 그러나 known issue는 impostor가 *더 밝다*고 보고. 이 모순은 Three의 BRDF normalization 어딘가에서 내가 놓친 factor가 있거나, off-peak 영역에서 Phong R-V vs BlinnPhong half-vector의 falloff curve 차이가 다른 곳에서 기여하기 때문.

**결론:** 측정 없이 fix하면 추측 fix. 위 모순 해소를 위해 webview parity test 인프라 (J1-Option B) 필요. 16.0에서 분리.

### J4. 빌드 시스템 / 패키지 변경

- `pixelmatch@7.x`는 ESM-only. esbuild가 CJS bundle 시 transpile (export default 정상 처리).
- `pngjs@7.x`는 CJS, native binding 없음 (순수 JS). Linux headless 환경에서 추가 의존성 제로.
- esbuild `harnessConfig`: format=cjs, external=['puppeteer']만, pixelmatch+pngjs 인라인. Output ~80 KB.
- `package-lock.json`은 평소 sync 안 되는 패턴 (이미 0.13.0에 멈춰 있음 — STATUS.md 노트 참조). 수정 안 함.
