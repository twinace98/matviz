## Autonomous judgments recorded

### J1. Single-frame fallback도 같은 entry로

`parseXsfTraj`는 ANIMSTEPS 없으면 `parseXsf` 호출 + wrap. → index.ts의
.xsf/.axsf dispatch 한 곳이 single + multi 모두 처리. 호출 사이트 단순화.

대안 (single은 parseXsf, multi는 parseXsfTraj 분기) 거부 — 호출 사이트가
multi 여부를 미리 알아야 함.

### J2. Lattice 객체 reference 공유 (fixed-cell 최적화)

`latticeMode='fixed'` 시 모든 frame이 같은 lattice array 객체를 share (deep
copy 안 함). renderer의 `setFrame`이 latticeMode='per-frame'일 때만
axisIndicator 업데이트하는 로직과 짝 (test J5에서 검증).

대안 (deep copy)는 안전성 ↑이지만 renderer 최적화 가능성 ↓. CrystalTrajectory
invariant에서 명시: "latticeMode='fixed' ⇒ frames[i].lattice === frames[0].lattice
(object reference equality)".

### J3. PRIMVEC numbered (per-frame) 처리

`PRIMVEC 1`, `PRIMVEC 2` 형태로 frame-indexed. 코드는 token[1]이 숫자면
per-frame, 아니면 shared로 분류. 파서 단위에서는 두 케이스 모두 정확 처리,
fixture는 fixed-cell case만 (variable-cell fixture는 17.1.2 XDATCAR NPT에서
강력하게 검증할 것).

### J4. Out-of-order PRIMCOORD 안전성

`PRIMCOORD k` 의 k가 1-based + 임의 순서 가능 가정. `frameAtoms[]`는 length가
부족하면 패딩하면서 frame index에 따라 채움. 일반 AXSF는 1, 2, 3, ... 순차이지만
robustness 추가.

### J5. 빈 frame 스킵

PRIMCOORD 없는 ANIMSTEPS 인덱스는 frames에 안 추가. AXSF 표준은 모든 frame이
PRIMCOORD를 가지지만 corrupt file에서 missing 가능 — silent skip + frames
length가 ANIMSTEPS와 다를 수 있음. Caller (renderer.loadTrajectory)는
실제 frames.length를 신뢰.

### J6. Volumetric 지원 (multi-frame은 first only)

AXSF가 BLOCK_DATAGRID_3D를 multi-frame trajectory와 함께 갖는 케이스는 매우
드뭄. 만약 있다면 첫 번째 grid만 파싱 + return (CLI/webview는 frames[0]에
연관). Plan §16 isosurface는 single-frame 가정.

## Known limitations / deferred

1. **Variable-cell AXSF fixture 부재** — 코드는 `PRIMVEC k` 처리 가능하지만
   fixture는 fixed-cell만. NPT case는 17.1.2 XDATCAR fixture로 강력 검증.
2. **Volumetric multi-frame** — 첫 frame만. 일반 AXSF에서 무관.
3. **PRIMCOORD 외 atom syntax (ATOMS 섹션)** — 17.1.0에서 single-frame
   parseXsf가 처리 (multi-frame ATOMS는 비표준).
4. **CLI 노출 없음** — CLI는 parseXsf 사용 (첫 frame만). v0.17.x trajectory
   CLI 진입 시 parseXsfTraj 사용 가능.
