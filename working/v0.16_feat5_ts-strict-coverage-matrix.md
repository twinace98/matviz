### J3. `noUncheckedIndexedAccess` deferred 근거 (핵심 결정)

949 에러의 구조:
- Array access (`arr[i]`) 후 undefined 가능성 narrow 필요. 950여 호출 사이트
  각각 `if (x !== undefined)` guard 또는 `arr[i]!` non-null assertion 추가.
- 작업량 추정: 반나절~1일. 산발적 fix — 대규모 일괄 patch로 가독성 악화 위험.
- 순익 평가:
  - **안전성 이득**: bounds-check NPE 방지. 그러나 현재 코드는 InstancedMesh
    loop가 정직 (`for (let i = 0; i < indices.length; i++)`) → out-of-bounds
    access가 사실상 발생 안 함. 이론적 안전성이지 실무 위험 낮음.
  - **가독성 손실**: 모든 array access에 guard 삽입 시 주 로직 흐름 분산.
    또는 `!` assertion 남발 → 타입 시스템 우회, strict 도입 원래 목적과 모순.

**선택: deferred to v0.16.x, 모듈별 점진 도입.** 순서 제안:
1. `src/parsers/*.ts` (입력 처리가 가장 NPE-prone)
2. `src/webview/math/symEigen.ts` + `wulff.ts` (수학 코드 안전성 이득 큼)
3. `src/webview/renderer.ts` (긴 파일, 마지막)

코드 전반 점진 적용 → 현재 v0.16 끝내기 전제조건 아님.
