## Autonomous judgments recorded

### J1. magneticArrowRenderer 코드 클로닝 vs 공유

magneticArrowRenderer와 displacementArrowRenderer는 구조 거의 동일
(cone+cylinder InstancedMesh, Quaternion 정렬, instanceColor). 차이점은
colormap (Red-Blue diverging vs Viridis-only)과 default scale.

**선택**: 별도 파일 클로닝. 근거:
- 한 renderer가 magnetic + displacement 둘 다 동시 표시 가능 (NiO에 변위 +
  자기 모멘트 둘 다 보이는 케이스).
- 공유 시 setColormap 등 mutable state로 두 path가 경쟁.
- 코드 ~140 LOC 두 곳 — diff로 구조 동기화 어렵지 않음 (둘 다 vertexColors
  함정 회피 주석 박혀 있음).

향후 리팩토링이 필요하면 v0.18+ 검토.

### J2. ComparisonSecondaryPhase는 ref 보존

`comparisonSecondaryPhase`가 `secondaryPhases[0]` 참조. addPhase로 새
phase 추가하면 array push되지만 `secondaryPhases[0]` 자체는 안 변함 → 기존
comparison은 첫 phase 유지.

clearPhases는 array 비우면서 active comparison도 해제 → 안전.

### J3. recomputeComparison는 setFrame 끝에서 호출

setFrame 끝에 NN matching + arrow rebuild. 30fps 재생 + 1k atoms에서 NN
N²=1M ops or N×27=27k ops (binning) — JS에서 ~ms. 30fps budget (33ms) 내
충분. 5k atoms에서는 auto-disable으로 차단 (compareToPhase가 reject).

### J4. compareToPhase return값으로 user-facing reason

명령 진입 시 reason 알리는 방식: webview console.warn (현재). vscode
notification 띄우려면 webview→extension 메시지 추가 필요. 첫 컷에서는
console.warn으로 충분 (사용자 측 Devtools 열어서 확인). v0.17.x에서 확장
가능.

### J5. clearPhases와 active comparison 결합 해제

명시적으로 `if (this.comparisonActive) this.clearComparison();` — clearPhases가
secondary 비우면 comparison이 dangling reference 갖게 됨 (object 자체는
GC 안 되지만 의미 없음). 대칭 상태 관리.

### J6. UI는 명령 팔레트만 (사이드 패널 부재)

Plan §17.2의 사이드 패널 phase UI도 deferred 였음. v0.17.1도 동일 패턴 —
명령 팔레트로 진입 + clear. 사이드 패널 통합은 v0.17.x에서 일괄 처리
권장 (Phases section + Comparison toggle을 함께 디자인).

### J7. PBC 무시 (working log §carry-over)

NPT XDATCAR에서 atom이 cell wrap하면 큰 변위 화살표 발생. 현 구현은 absolute
cartesian만 비교. 사용자가 wrap-aware 비교 원하면 v0.17.x에 minimum-image
distance 옵션 추가.
