## Autonomous judgments recorded

### J1. Wulff vertex 누적 후 ConvexGeometry — 16.4 webview와 동일

src/webview/wulff.ts와 동일한 알고리즘. Webview는 `wulff.ts` 모듈로 분리,
CLI는 inline 함수로 재현. 모두 같은 idea: enumerate triples → solve →
test → ConvexGeometry. 하나의 자연스러운 매핑 — webview의 `verts`
누적이 그대로 CLI의 `verts.push(v)` 으로.

### J2. ConvexGeometry는 render-helpers에서 이미 import

`scripts/render.ts:201`이 `import { ..., ConvexGeometry } from '${helpersURL}'`
로 ConvexGeometry를 이미 import. → 추가 작업 없이 inline 사용 가능. 이게
16.4가 가장 마지막에 포팅되어도 작업량 적은 이유 (이미 helpers에 ConvexGeometry
가 있음 = v0.15.2 다면체 작업의 잔여 인프라 재사용).

### J3. Symmetry expansion 미적용 (webview와 동일)

Webview J3과 일관 — 사용자가 모든 face explicit 입력. cubic {100} = 6, {111}
= 8을 따로 listing. SKILL.md에 명시.

→ 16.x deferred (cubic system heuristic으로 자동 expansion).

### J4. View=std + γ=1.5 fixture

NaCl 5.64Å cell + γ=1.5 → Wulff 면 거리 ~1.5Å → polytope 한 변 ~3Å — 셀
한 변의 절반 정도, 시각적으로 무난.

view=std는 표준 결정 view (c-axis up, view from a*). Polytope가 origin에
위치하므로 셀 한쪽 가장자리에 보임 — 회귀 catching 목적엔 충분.

### J5. Bounding-box fallback 그대로 포팅

webview wulff.ts의 fallback 로직 inline 재현. user input이 부정확하거나
under-constrained면 bbox로 bounded — CLI에서도 동일 거동.

## Known limitations / deferred (CLI 4 ports 누적)

이전 webview limitations carry-over (각 부모 working log 참조):
- Symmetry-op rotation R·U·Rᵀ (16.1 + 16.3) — non-P1 + off-diagonal 부정확.

- Compressed MAGMOM (16.3) — reject + warn.
- Aniso + occupancy<1 transparent ellipsoid (16.1 + 16.2 교집합).
- Symmetry plane expansion for Wulff (16.4) — explicit listing required.
- Custom χ²₃ contour (16.1) — 0.5/0.9만.

- `--wulff` 입력 syntax는 한 줄로만 (multi-line 입력 어려움). preset
