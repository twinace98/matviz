# v0.16 — Extended Crystallography

**목표:** 결정학적 고급 속성을 시각화. anisotropic displacement parameter (thermal ellipsoid), partial occupancy, magnetic moment vector, crystal morphology (Wulff). VESTA가 표시하는 핵심 속성을 같은 수준의 정확도로 재현하는 것이 지표. 부수적으로 Plan.md `0269d61`이 정의한 v0.15.1 작업(impostor 톤맵 + visual-regression harness)을 16.0으로 흡수하여 v0.15 잔여 known issue를 닫는다.

**상태:** 킥오프 초안 — 승인 후 **완전 자율 진행** (체크인 없음).

## 피처 간 의존성

- **16.0 먼저** — visual-regression harness가 16.1 ellipsoid의 정량적 검증(probability contour 50%/90% 부피 비율, eigen 수치 안정성)에 *전제조건*. impostor 톤맵 fix도 같이 닫아 v0.15 known issue 잔여 0.
- **16.1 → 16.2** — 둘 다 CIF 파서 + per-instance attribute 확장. 같은 InstancedMesh 분기 path를 손대므로 순차가 효율. 16.1에서 `cifParser.ts`를 크게 손대므로 NaN guards (다른 기술부채 항목) 동시 처리.
- **16.3 독립** — 새 화살표 InstancedMesh, 기존 atom 경로와 격리. 16.1/16.2 후 진입.
- **16.4 격리** — 새 Wulff 렌더 컴포넌트, 명령 팔레트 진입. 다른 피처와 의존성 없음. **scope 압박 시 v0.16.1로 자율 분리 가능** (탈출 밸브).
- **16.5 마지막** — 모든 파서 변경(16.1/16.2/16.3) 후 일괄 strict mode 적용이 효율. 신규 옵셔널 필드들이 strict-safe 한지 일괄 검증.

**제안 순서:** 16.0 → 16.1 → 16.2 → 16.3 → 16.4 → 16.5.

## 결정 게이트 (사전 채택 — 자율 진행 중 별도 사용자 승인 불필요)

자율 루프 정책상 stop trigger #2 (decision gate)를 회피하기 위해 plan 시점에 모든 분기 결정을 확정:

1. **16.1 Ellipsoid 지오메트리 디스패치**: InstancedMesh + per-instance 4×4 매트릭스 (unit-radius sphere geometry를 ellipsoid로 변형). impostor 셰이더는 uniform radius 가정이므로 **ellipsoid 사이트는 Phong 경로 강제** — `forceEllipsoidPhong = true` 분기를 atom render 진입에 추가. ellipsoid 전용 impostor 셰이더는 v0.17+ deferred.
2. **16.1 Probability contour 스케일**: χ²₃ 분포 기반 하드코드 — `CONTOUR_50PCT_SCALE = sqrt(2.366)`, `CONTOUR_90PCT_SCALE = sqrt(6.251)`. erfc 라이브러리 도입 안 함.
3. **16.1 Eigen solver**: 3×3 symmetric Jacobi rotation, 새 파일 `src/webview/math/symEigen.ts` (~30 LOC). 외부 의존성 추가 안 함.
4. **16.2 Partial occupancy 렌더 방식**: stacked transparent sphere (단순). sectored-sphere 셰이더는 deferred.
5. **16.3 MAGMOM 파일 디스커버리**: POSCAR 로드 시 같은 디렉토리 `INCAR` 또는 `OUTCAR` 자동 검색 → MAGMOM 라인 파싱 → `magMom` 주입. 발견 시 적용, 미발견 시 침묵.
6. **16.3 Multi-frame scope**: v0.16에서는 single-frame magnetic moment만. trajectory 평균 또는 frame별 표시는 v0.17 `CrystalTrajectory`.
7. **16.4 Wulff 알고리즘**: 직접 구현 — 반-공간 교차 (Sutherland-Hodgman 변형) ~80 LOC, 새 파일 `src/webview/wulff.ts`. 라이브러리 도입 안 함.
8. **16.5 TS strict 적용 범위**: `strict: true` + `noUncheckedIndexedAccess`. `noPropertyAccessFromIndexSignature`는 보류 (overhead 큼).
9. **16.4 Scope-pressure deferred 트리거**: 16.1 또는 16.3 완료 시점에 누적 LOC > 2000 또는 작업일 견적 > 7일이면 **자율 결정으로 v0.16.1 분리** 후 v0.16.0 릴리즈. 별도 사용자 승인 불필요 (사전 위임).

### 로컬 commit 정책 (사용자 명시 지시 2026-04-22)

각 .x 피처(16.0, 16.1, …, 16.5)는 완료 시 **최소 1개의 checkpoint commit**을 자율로 생성. 사용자가 나중에 임의 .x 시점으로 `git checkout`/`git revert`로 돌아갈 수 있도록 분리 보존이 목적. 이 commit 권한은 사전 위임 — 별도 승인 불필요.

**Commit 시점 (순서 엄수):**
1. 해당 .x의 모든 통과 기준 만족.
2. `npx tsc --noEmit` → `npm run build` → (16.0 이후) `npm run test:visual` → manual sanity 통과.
3. `working/v0.16_feat{N}_*.md` 작성 완료.
4. `git add` (소스 + working log + 신규 fixture) → `git commit` (메시지 가이드라인 따름).
5. 다음 .x 피처로 즉시 진입.

검증 (2번 단계) 실패 시 commit 보류 + root cause fix → 재검증 → commit.

**Commit 단위:**
- **각 .x = 최소 1 commit** (체크포인트 보장 — 사용자가 .x 단위로 git history 탐색 가능).
- 한 .x 안에서 자연스럽게 분리되는 작업(예: 16.1의 "파서 확장" / "symEigen" / "ellipsoidRenderer + UI")은 **자율로 sub-commit 분할 가능**. 모든 sub-commit이 *각자* 검증 통과 상태여야 하며 (bisect 가능), 마지막 sub-commit이 .x의 checkpoint 역할.
- 분할 시 commit 메시지에 일련 번호 명시 (예: `(1/3)`, `(2/3)`, `(3/3)`).

**Pre-commit hook 실패:**
- `--no-verify` 사용 금지 (CLAUDE.md 규칙 그대로). hook 실패 시 root cause fix 후 *NEW* commit 생성 (`--amend` 사용 안 함; 이전 commit이 hook 실패로 commit 안 됐기 때문).
- 자명한 fix가 없으면 stop trigger #1 (build failure) 적용.

**Push 정책:**
- 로컬 commit은 사전 위임. **`git push origin main`, `git push --tags`, `git push sogang-qmp`는 여전히 사전 사용자 승인.**
- 자율 진행 중에는 모든 commit이 로컬에만 누적. 사용자가 push를 명시하면 그때 일괄 push.

## Commit 메시지 가이드라인

기존 release commit 패턴(v0.13.1, v0.14, v0.15)과 일관성 유지하되, 각 .x별 분리에 맞춰 더 좁은 scope로 작성. 언어는 **영어** (기존 git history 일관성). 본문에서 한국어가 자연스러운 부분(설계 근거 등)은 mix 가능. 사용자가 한국어 commit 요청 시 즉시 전환.

**제목 (50–70자 권장, 100자 이내):**
```
feat(v0.16.{N}): {한 줄 피처 요약}
```

예시:
- `feat(v0.16.0): visual-regression harness + impostor sRGB tone fix`
- `feat(v0.16.1): CIF aniso parser, 3x3 sym eigen, ellipsoid renderer`
- `feat(v0.16.2): partial occupancy via stacked transparent spheres`
- `feat(v0.16.3): VASP/CIF magnetic moment vectors`
- `feat(v0.16.4): Wulff construction via half-space intersection`
- `feat(v0.16.5): TS strict mode + parser fixture coverage matrix`

Sub-commit 분할 시:
- `feat(v0.16.1, 1/3): CIF _atom_site_aniso_* parser + B-form conversion`
- `feat(v0.16.1, 2/3): symmetric 3x3 Jacobi eigendecomposition`
- `feat(v0.16.1, 3/3): ellipsoid InstancedMesh + probability contour UI`

**본문 구성 (sub-section 헤더 필수):**

```
{1~2문장 context: 왜 이 변경, 무엇을 닫음}

Changes:
- {file or module}: {what changed, one line per item}
- {file}: {...}
- {...}

Design decisions:
- {결정 1, 다른 선택지를 거부한 근거 포함}
- {결정 2}

Verification:
- npx tsc --noEmit: 0 errors (또는 strict 적용 후 N → 0)
- npm run build: pass
- npm run test:visual: ΔRGB max=X, mean=X, p95=X (16.0 이후)
- manual: {fixture}에서 {확인한 동작 한 줄}

Known limitations / deferred:
- {알려진 한계 1}
- {deferred 항목 (다음 .x 또는 v0.17+로 미룬 것)}

Log: working/v0.16_feat{N}_*.md

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```

**왜 이 양식인가:**
- 사용자가 1년 후 `git log -p` / `git show <hash>`로 들어왔을 때 *그 한 commit만으로* 무엇을 왜 바꿨는지 이해 가능해야 함.
- "Changes" = 무엇 / "Design decisions" = 왜 / "Verification" = 어떻게 검증 / "Known limitations" = 무엇이 안 됨 → 회귀 추적·복원 의사결정에 필요한 4가지 정보 보장.
- "Log:" cross-reference로 working log의 자세한 시도/실패 기록과 commit message의 결과 요약을 분리 — commit history가 검색 가능한 인덱스, working log가 상세 본문 역할.
