## 0. CrystalStructure 확장 (전 피처 공통 전제)

**파일:** `src/parsers/types.ts`

기존 8개 필드는 건드리지 않고 옵셔널 필드만 추가:

```ts
export interface CrystalStructure {
  // 기존 필드 (lattice, species, positions, pbc, title, spaceGroup, cellParams, symmetryOps) — 무수정.
  
  // v0.16 신규 (모두 옵셔널 — 미파싱 시 undefined; 기존 파서/렌더 경로 변동 0)
  thermalAniso?: Array<{ U11: number; U22: number; U33: number; U12: number; U13: number; U23: number } | null>;
  occupancy?: number[];                      // per-atom, default 1.0
  magMom?: Array<[number, number, number]>;  // per-atom Cartesian moment vector (μB)
}
```

**왜 옵셔널 + per-atom array**: 부분 점유 사이트는 동일 좌표에 여러 종이 존재. `species`/`positions`/`occupancy`가 같은 길이로 정렬되며, occupancy 합 ≤ 1.0인 사이트는 stacked rendering. ellipsoid가 없는 원자는 `null`로 마킹하여 길이 정렬 유지.

**Webview 메시지**: `src/webview/message.ts` 변경 없음 — `CrystalStructure` 자체가 postMessage로 전달되므로 옵셔널 필드는 자동 전달. 웹뷰 측은 `data.thermalAniso?.[i]` 식으로 안전 접근.

## 16.1 — Thermal ellipsoids + parser NaN guards

**파일:**
- 수정 `src/parsers/cifParser.ts`: `_atom_site_aniso_label` 별도 루프 + `_atom_site_aniso_U_*` 컬럼 6개 (또는 `_atom_site_aniso_B_*` → U-form 변환); degenerate lattice (γ=0/180, a/b/c=0) NaN guard.
- 수정 `src/parsers/pdbParser.ts`: 동일 NaN guard (`pdbParser.ts:66` 부근).
- 수정 `src/parsers/types.ts`: `thermalAniso?` 필드 (위 0번 섹션).
- 신규 `src/webview/math/symEigen.ts`: 3×3 symmetric Jacobi rotation (~30 LOC).
- 신규 `src/webview/ellipsoidRenderer.ts`: per-instance 4×4 매트릭스로 unit sphere를 ellipsoid 변형. `MeshPhongMaterial` 강제.
- 수정 `src/webview/renderer.ts`: ellipsoid 사이트는 Phong 경로 강제; ellipsoid 토글 / probability contour state.
- 수정 `src/webview/main.ts`: 사이드 패널 "Thermal ellipsoids" 섹션 (토글 + 50% / 90% 라디오).

**CIF aniso 파싱 데이터 플로우:**
1. CIF에는 두 개의 site 루프가 가능: `_atom_site_*` (좌표) + `_atom_site_aniso_*` (anisotropic). 후자는 별도 loop_ 블록.
2. `_atom_site_aniso_label`이 `_atom_site_label`과 매칭됨. 매칭 인덱스로 `thermalAniso[i]` 채움.
3. 6 컴포넌트: `_atom_site_aniso_U_11`, `U_22`, `U_33`, `U_12`, `U_13`, `U_23` (Å²). 일부 CIF는 B-form 사용 — `_atom_site_aniso_B_*` 발견 시 `U = B / (8π²)` 변환.
4. 미매칭 사이트는 `null`. anisotropic 루프 자체가 없으면 `thermalAniso` 필드 자체 omit.

**Eigen → ellipsoid 변환:**
1. Symmetric tensor `M = [[U11, U12, U13], [U12, U22, U23], [U13, U23, U33]]` (Å²).
2. Jacobi rotation → eigenvalues `λ₁, λ₂, λ₃` (≥0; 음수 시 0으로 클램프 + 콘솔 warning) + 정규직교 eigenvector matrix `R` (열 = eigenvector).
3. 반축: `rᵢ = scale × sqrt(λᵢ)`, where `scale = sqrt(2.366)` (50%) 또는 `sqrt(6.251)` (90%) — 하드코드 χ²₃ 표.
4. Per-instance 4×4 매트릭스: `T(positionᵢ) × R4 × diag(r₁, r₂, r₃, 1)`, R4는 R을 4×4로 확장 (마지막 행/열 = 0,0,0,1).

**`symEigen.ts` 인터페이스:**
```ts
export function jacobiSym3(M: [[number,number,number],[number,number,number],[number,number,number]]):
  { values: [number,number,number]; vectors: [[number,number,number],[number,number,number],[number,number,number]] };
```
- 표준 Jacobi: 가장 큰 off-diagonal element → Givens rotation으로 0화 → 반복. 수렴 임계값 1e-10, max iter 50.
- vectors[i]는 i번째 eigenvector (열 형태). values와 인덱스 일치.

**Ellipsoid InstancedMesh 디스패치:**
1. atom 빌드 시 `thermalAniso?.[i] != null && showEllipsoids` 인 사이트는 `ellipsoidGroup`로, 나머지는 기존 sphere/impostor 그룹으로 분리.
2. Ellipsoid 그룹: unit-radius `SphereGeometry` (32-segment) + per-instance `Matrix4`. `MeshPhongMaterial` 사용 (impostor 토글 무시).
3. Per-element color는 기존 atom color 경로 재사용 (instance color attribute).
4. Selection highlight (`HIGHLIGHT_COLOR`)는 ellipsoid 그룹에도 동작 — material color 변경 path 동일.

**NaN guards (CIF/PDB):**
1. `cifParser.ts` cell length/angle 변환부 (`a*sin(γ)`, `c/sin(γ)` 등): 분모가 `< 1e-6` 이면 `throw new Error("Degenerate lattice: angle ≈ 0 or 180, cannot build cell vectors")`.
2. `pdbParser.ts:66` CRYST1: 동일 가드.
3. 두 파서 모두 throw로 위임 — `crystalEditorProvider.ts`의 v0.13.1 error boundary가 받아 "Open as Text" 토스트 표시.

**통과 기준:**
- `test/fixtures/calcite-aniso.cif` (또는 anorthite) 로드 시 Ca/Mg/C 사이트에 ellipsoid 표시; VESTA 같은 fixture 스크린샷과 시각 비교 일치.
- 50% / 90% 토글 시 ellipsoid 부피가 `(6.251/2.366)^1.5 ≈ 4.30×` 차이 (육안 + 픽셀 카운트로 sanity).
- Degenerate lattice fixture (γ=0) 로드 시 "Open as Text" 토스트 정상.
- 시각 회귀 (16.0 harness): ellipsoid 사이트는 Phong 경로이므로 baseline 갱신 시 수치 안정. ellipsoid가 없는 fixture는 ΔRGB 영향 0.

**작업 단위 추정:** 2.5일 (CIF 파서 확장 1일 + symEigen 0.5일 + ellipsoidRenderer 1일).

## 16.2 — Partial occupancy

**파일:**
- 수정 `src/parsers/cifParser.ts`: `_atom_site_occupancy` 컬럼 파싱.
- 수정 `src/parsers/types.ts`: `occupancy?` 필드 (이미 0번).
- 수정 `src/webview/renderer.ts`: 부분 점유 사이트는 stacked transparent sphere; 사이드 패널 토글.
- 수정 `src/webview/main.ts`: 사이드 패널에 "Partial occupancy" 토글 (default off).

**Stacked transparent sphere 데이터 플로우:**
1. CIF 파싱 시 동일 (또는 < 0.1 Å 거리) 좌표 + 다른 species + occupancy < 1.0 엔트리들을 하나의 *site cluster*로 그룹화. `species`/`positions`/`occupancy`는 그대로 평탄 배열 유지 (그룹화는 렌더링 시점에).
2. `showPartialOccupancy && occupancy?.[i] < 1.0` 인 사이트:
   - 같은 element 내에서 InstancedMesh entry 추가; material에 `transparent: true, opacity: occupancyᵢ` 적용.
   - Z-fighting 방지: 같은 사이트의 k번째 species는 sphere 반지름을 `0.99^k × baseRadius`로 미세 축소.
3. `showPartialOccupancy = false` 시 (default): 같은 site cluster에서 가장 큰 occupancy 종만 full opacity로 표시 (기존 거동 보존; 시각 회귀 0).
4. Bond detection은 occupancy 무시 — 가장 큰 occupancy 종 기준으로 단일 atom처럼 처리. Ellipsoid (16.1)와 occupancy의 상호작용: ellipsoid는 transparent stacking이 아니라 자체 opacity 무시 (single ellipsoid per site).

**Occupancy 정규화 정책:**
- 합 > 1.0인 사이트 (드물지만 disordered 광물에서 발생) → raw 사용 (사용자 가시 — 모델 자체의 비물리적 특성을 숨기지 않음).
- 합 < 1.0인 사이트 → 빈 자리 (vacancy)는 시각화 안 함.

**통과 기준:**
- `test/fixtures/mineral-mix.cif` (예: `(Fe,Mg)SiO3` orthopyroxene) 로드 + 토글 on 시 혼합 사이트가 비율로 시각화 (Fe 큰 sphere + Mg 작은 sphere stacked).
- 토글 off 시 가장 큰 occupancy 종만 표시 — `test/fixtures/nacl.cif` 같은 단일 점유 fixture는 ΔRGB 영향 0.
- 시각 회귀 (16.0): default off이므로 기존 fixture 변동 없음.

**작업 단위 추정:** 1.5일.

## 16.3 — Magnetic moment vectors

**파일:**
- 수정 `src/parsers/poscarParser.ts`: POSCAR 1번째 코멘트 라인의 `MAGMOM = ...` 추출 (선택적).
- 수정 `src/extension.ts` 또는 `src/editor/crystalEditorProvider.ts`: POSCAR 로드 시 같은 디렉토리 `INCAR` 파일 검색 → `MAGMOM` 라인 추출 → `magMom` 배열로 주입.
- 수정 `src/parsers/cifParser.ts`: `_atom_site_moment_label` 별도 루프 + `_atom_site_moment_crystalaxis_x/y/z` 또는 `_atom_site_moment_cartn_x/y/z`. crystalaxis 형식이면 lattice basis로 cartesian 변환.
- 수정 `src/parsers/types.ts`: `magMom?` 필드 (이미 0번).
- 신규 `src/webview/magneticArrowRenderer.ts`: arrow InstancedMesh (cone tip + cylinder shaft, 둘 다 instanced).
- 수정 `src/webview/main.ts`: 사이드 패널 "Magnetic moments" 토글 + colormap 셀렉터 (default: red-blue diverging).

**Arrow InstancedMesh 구조:**
1. **Shaft**: thin `CylinderGeometry` (radius 0.05 Å, length 1, 8-segment). Per-instance 4×4 매트릭스로 위치(원자 중심)/방향(moment normalize)/스케일 (length = `|m| × scale`, default scale = 1.0 Å/μB).
2. **Tip**: `ConeGeometry` (radius 0.15 Å, height 0.3 Å). 같은 방향, shaft 끝에 위치.
3. **Color**: `colormap(|m|)` per-instance attribute로 fragment 셰이더에 전달. 기본 colormap: red(+)–white(0)–blue(−) diverging. 사용자 셀렉터: viridis, plasma, jet (간단한 lookup table).
4. 0 moment 사이트 (`|m| < 1e-6`)는 instance 자체 생략 (per-instance count에서 제외).

**MAGMOM 파싱 (VASP):**
1. **POSCAR 코멘트 라인 우선**: 1번째 라인에 `MAGMOM = ...` 패턴 있으면 사용. 형식: 공백 구분 floats (collinear) 또는 3-vector 그룹 (non-collinear).
2. **인접 INCAR fallback**: POSCAR와 같은 디렉토리에 `INCAR` 또는 `OUTCAR` 검색. INCAR 우선. `MAGMOM = 1.0 -1.0 0 0 ...` 라인 추출.
3. **Collinear vs non-collinear 판별**: MAGMOM 토큰 수 = N atoms → collinear (`[0, 0, magmom_i]`로 z-axis 가정); 토큰 수 = 3N → non-collinear vector. 그 외 토큰 수 → 침묵 무시 + 콘솔 warning.
4. **VASP 압축 표기 (`4*1.0 -2*1.5`) 미지원**: 첫 컷에서는 풀어 쓴 형식만 (압축 표기는 v0.16.1로 deferred — 단순 정규식으로 N개 atom과 매칭 불일치 시 무시).
5. **CIF moment**: `_atom_site_moment_label`로 매칭. `crystalaxis` 형식은 fractional → cartesian 변환.

**Multi-frame scope 결정:**
v0.16에서는 single-frame만. POSCAR/CIF의 MAGMOM은 정적 — MD trajectory의 frame별 MAGMOM은 v0.17 `CrystalTrajectory` shape에 같이 처리.

**통과 기준:**
- `test/fixtures/nio-afm/POSCAR` + `INCAR` (`MAGMOM = 2 -2 0 0`) 로드 시 Ni 사이트에 ↑↓ 화살표 alternating, O 사이트는 0 (화살표 없음).
- `test/fixtures/fe-fm/POSCAR` + `INCAR` (`MAGMOM = 2.2 2.2`) 로드 시 두 Fe 모두 같은 방향.
- 토글 off 시 화살표 사라짐, 시각 회귀 (16.0) 무영향.
- INCAR 미존재 시 침묵 (콘솔 warning만).

## 16.4 — Wulff construction (scope-pressure deferrable)

**파일:**
- 신규 `src/webview/wulff.ts`: 반-공간 교차 알고리즘 (~80 LOC).
- 수정 `src/webview/renderer.ts`: Wulff 다면체 렌더 (별도 `wulffGroup`, transparent material, default opacity 0.5).
- 수정 `src/extension.ts`: 명령 팔레트 `MatViz: Show Wulff Construction` 등록. webview 내 입력 모달 또는 VSCode `showInputBox` 시퀀스로 (hkl) + γ 페어 리스트 입력.
- 수정 `src/webview/main.ts`: 사이드 패널 "Wulff" 토글 + opacity 슬라이더 + edge-color 셀렉터.

**반-공간 교차 알고리즘:**
1. 입력: `[(h, k, l, γ)]` 리스트 (γ = surface energy, 임의 단위; 비율만 의미 있음).
2. 각 (hkl) → reciprocal-lattice direction `n_hkl = h·a* + k·b* + l·c*`, 정규화.
3. Wulff theorem: 각 face plane은 origin으로부터 distance `d_hkl ∝ γ_hkl` (정규화된 distance).
4. 시작 polytope: 큰 cube (한 변 = `10 × max(d_hkl)`).
5. 각 평면 `(n_hkl, d_hkl)`에 대해 polytope를 half-space `n·x ≤ d`로 clipping:
   - Vertex 리스트 + face 리스트 표현.
   - 각 face마다 vertex가 평면 양쪽에 분포하면 새 vertex 생성 (선분 교차) + face 분할.
   - 평면 너머 vertex는 제거.
   - 새 face (clipped edges 모음) 추가.
6. 결과: 각 face polygon → fan triangulation → `BufferGeometry`.
7. 중심을 결정 cell origin에 고정. opacity는 사이드 패널에서 조절.

**알고리즘 단순화:** Sutherland-Hodgman 변형이지만 3D면이라 까다로움 — 각 평면 적용을 *polytope half-space clip* 한 함수로 추상화. 라이브러리 (`@thi.ng/geom-hull`) 미도입 결정은 의존성 추가 비용 회피 + 단순 케이스(Au fcc 큐빅) 충분 검증 가능.

**UI 데이터 플로우:**
1. 명령 팔레트 진입 → VSCode `showInputBox` 다중 입력: "h,k,l,γ"  형식 리스트 (예: `1,0,0,1.0; 1,1,1,1.15`).
2. 파싱 후 webview에 `{ type: 'wulff', planes: [...] }` postMessage.
3. webview가 `wulff.ts` 호출 → polytope 생성 → `wulffGroup`에 추가 + 사이드 패널 토글 자동 on.

**통과 기준:**
- Au fcc 입력 `(1,0,0)=1.0, (1,1,1)=1.15` 결과가 cuboctahedron 변형 (관측 일치, 사진과 비교).
- 단순 cube 입력 `(1,0,0)=1.0` 단독 시 cube 출력.

**Scope-pressure deferred 트리거 (사전 위임):**

## 전 피처 공통 검증 + checkpoint commit

각 .x 피처 완료 시 다음 순서를 엄수:

1. **타입체크**: `npx tsc --noEmit` (16.5 이전엔 strict 미적용 상태에서 zero errors).
2. **빌드**: `npm run build`.
3. **시각 회귀**: `npm run test:visual` (16.0 완료 후 모든 피처에서 필수).
4. **Manual sanity**: 해당 피처 fixture 로드 → 토글 동작 → 시각적 회귀 무 (가능하면 다중 fixture).
5. **Working log 작성**: `working/v0.16_feat{N}_*.md` — 변경 요약, 견적/실측 표, 시각 sanity 노트, 알려진 한계.
6. **Stage**: `git status` 확인 → `git add` 관련 파일들 (소스 + working log + 신규 fixture).
7. **Commit**: 메시지 가이드라인(version plan)에 따라 `git commit`. heredoc 사용 (멀티라인 본문).
8. 다음 .x 피처로 즉시 진입 (또는 16.4 분리 트리거 발동 시 v0.16.0 릴리즈 사전 승인 요청).

검증 (1~4) 실패 시 commit 보류 + root cause fix → 재검증 → commit. Pre-commit hook 실패 시 fix → NEW commit (`--amend` 금지 — CLAUDE.md 규칙).

**Sub-commit 분할 가이드:** 한 .x 안에서 자연스럽게 분리되는 작업 단위는 자율 분할 가능. 단 *각* sub-commit이 위 검증 1~4를 통과해야 함 (bisect 안전성). 예시:
- 16.1 → `(1/3)` CIF aniso 파서 + NaN guards / `(2/3)` symEigen / `(3/3)` ellipsoidRenderer + UI
- 16.0 → `(1/2)` impostor 셰이더 fix / `(2/2)` harness + baseline
- 그 외 .x는 단일 commit이 일반적.

**Commit 예시 (16.0):**
```
feat(v0.16.0): visual-regression harness + impostor sRGB tone fix

Closes the v0.15 known issue (impostor brighter than Phong) and lands
the pixelmatch-based regression harness used as the verification gate
for all subsequent v0.16 features.

Changes:
- src/webview/sphereImpostor.ts: remove manual linearToSRGB call;
  WebGLRenderer's outputColorSpace=sRGB encode is sufficient.
- src/webview/cylinderImpostor.ts: same fix, mirror sphere path.
- scripts/render.ts: add --impostor on|off|auto flag (auto = default,
  unchanged behavior). Inline HTML shader sources mirror webview fix.
- test/visual/harness.ts (new): pixelmatch-based PNG diff CLI; per
  fixture renders with --impostor on/off, computes max/mean/p95 ΔRGB,
  emits diff PNG to test/visual/diff/, exits 0 on gate pass.
- test/visual/fixtures/{nacl,silicon-2x2x2,diamond-16k,multielement}/
  scene.json (new): 4 fixture scene configs.
- test/visual/baseline/*.png (new): 8 baseline PNGs (4 fixtures × 2
  impostor states), generated via --update-baseline first run.
- package.json: add pixelmatch + pngjs devDeps; add scripts.test:visual.

Design decisions:
- Manual linearToSRGB removal vs adding tonemapping_fragment chunk:
  diagnose first (Plan.md v0.15.1.1) — root cause was double encoding
  from manual call, so removing it is the minimal fix. Chunk approach
  rejected (would change downstream behavior unrelated to this bug).
- pixelmatch + pngjs vs canvas/sharp: lighter dependency footprint,
  no native bindings, runs in headless Linux without extra libs.
- Gate thresholds (p95<2, mean<0.5): from Plan.md v0.15.1; loose
  enough to absorb anti-aliasing noise, tight enough to catch real
  shading drift.

Verification:
- npx tsc --noEmit: 0 errors
- npm run build: pass
- npm run test:visual: ΔRGB max=1.7, mean=0.31, p95=1.4 (all fixtures
  under gate). Before fix: max=14.2, mean=4.1, p95=11.8.
- manual: NaCl + 16k diamond rendered with impostor on/off — visually
  indistinguishable.

Known limitations / deferred:
- Baseline PNGs require regen if VSCode webview tone or Three.js
  default lighting changes. Document in working log.
- Harness only diffs PNGs; doesn't test interactivity (selection,
  measurement). Out of scope for v0.16.

Log: working/v0.16_feat0_visual-regression-harness.md

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
```

이 양식을 16.1~16.5에서도 동일하게 적용. Sub-commit은 같은 골격을 더 좁은 scope로 작성.
