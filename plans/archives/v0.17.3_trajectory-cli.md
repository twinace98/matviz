# v0.17.3 — Trajectory CLI Exposure

**목표:** v0.17.0/v0.17.2의 webview-only trajectory 기능을 headless CLI 렌더러
에 노출. 단일 frame 추출 (`--frame N`) + 이미지 시퀀스 렌더 (`--all-frames`)
2가지 모드. matviz-render skill을 통한 보고서 워크플로우에서 MD 결과 시각화
자동화 가능 (Claude가 trajectory PNG 시퀀스를 직접 생성 → ffmpeg/PIL로 gif/mp4
변환 또는 key frame 추출).

**전제 컨텍스트:** v0.17.2 종료 시점 main에서 시작. 묶음 릴리즈는 v0.17.3
closure 시점까지 보류 (옵션 A 그대로). v0.17.3 commits 누적 후 묶음 릴리즈에
합류.

**상태:** 킥오프 초안 — **즉시 자율 진행** (사용자 4 결정 carry-over).

## 결정 게이트 (사전 채택)

7개 게이트:

1. **Frame 인덱스 0-indexed**: 내부 일관성 (renderer.setFrame 등). 사용자는
   1-indexed에 익숙하지만 CLI는 프로그래밍 도구라 0-indexed 우선.
2. **Default `--frame 0`**: 현재 동작 (trajectory의 첫 frame 렌더). 비-
   trajectory 파일은 영향 0.
3. **Out-of-range clamp**: N < 0 → 0, N >= frameCount → frameCount-1.
   warn 메시지로 안내.
4. **`--all-frames` 우선**: `--frame N`과 충돌 시 `--all-frames` 실행 +
   warn. user가 단일 frame만 원하면 `--frame N` 단독 사용.
5. **Filename templating**: `<base>_<NNNN>.png` (4자리 zero-padded). 9999
   frame 초과는 `_NNNNN` 5자리 자동 확장. 출력 디렉토리는 `-o`의 디렉토리
   부분.
6. **Volumetric CLI는 first frame만**: trajectory + volumetric 결합은 v0.17
   범위 외 (Plan §16 isosurface는 single-frame 가정). 17.3.1/17.3.2는
   atom rendering만 frame별, volumetric 무시.
7. **Browser instance 재사용 (perf)**: `--all-frames`에서 매 frame마다 새
   Puppeteer 인스턴스 생성하면 100 frame이 ~3분. 한 번 생성 + 매 frame
   page.evaluate로 structure 교체 + screenshot. 5초 → 20초 (4× 가속).
   초기 implementation 단순성 위해 J7는 v0.17.x로 deferred — 17.3.2 첫
   컷은 매 frame 새 invocation (subprocess)으로 단순.
