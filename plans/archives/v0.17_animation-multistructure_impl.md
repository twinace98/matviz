## 0. CrystalTrajectory 타입 + 보조 API (전 피처 공통 전제)

**파일:** `src/parsers/types.ts`, `src/parsers/index.ts`

신규 타입 + parser 보조 entry:

```ts
// types.ts
export interface CrystalTrajectory {
  frames: CrystalStructure[];        // length ≥ 1
  latticeMode: 'fixed' | 'per-frame';
  // Convenience accessors (computed in parser, immutable):
  // - frames[0] is always the "first frame" — UI default before scrub
  // - For 'fixed' lattice mode, frames[i].lattice === frames[0].lattice (ref equal)
  //   so renderer's cell wireframe doesn't rebuild per frame
}
```

```ts
// index.ts — append (do NOT modify existing parseStructureFile)
export function parseStructureFileTraj(path: string, content: string): CrystalTrajectory {
  // Format dispatch by extension/content (mirrors parseStructureFile).
  // Each format-specific parser may emit:
  //   - CrystalStructure (single-frame) → wrapped to 1-frame trajectory
  //   - CrystalTrajectory directly (AXSF, XDATCAR, extended XYZ multi-frame)
  // Wrapping logic: { frames: [single], latticeMode: 'fixed' }.
}
```

**왜 별도 entry**: 기존 `parseStructureFile()`을 모든 호출 사이트가 import 중
(`extension.ts`, `crystalEditorProvider.ts`, `scripts/render.ts`, `scripts/test-aniso.ts`,
…). 시그니처 변경은 광범위 영향. 별도 함수 추가가 minimal-touch.

**호출 사이트 변경:**
- `crystalEditorProvider.ts`: 기존 `parseStructureFile` 호출 유지 + 새 trajectory
  detection (frames > 1) 시 `parseStructureFileTraj` 추가 호출 + webview에 전송.
- 기타 호출 사이트(`scripts/render.ts` 등): 변동 없음. CLI 렌더는 single-frame
  유지 (Plan: trajectory CLI 노출은 v0.17.x).

## 17.1.1 — AXSF (multi-frame XSF) 파서

**파일:**
- `src/parsers/xsfParser.ts` — `parseXsfTraj()` 추가 (single-frame은 기존
  `parseXsf()` 위임).
- 신규 fixture: `test/fixtures/test-trajectory.axsf` (4 frames, fixed lattice).

**AXSF 형식:**
```
ANIMSTEPS 4
PRIMVEC
  ax ay az
  bx by bz
  cx cy cz
PRIMCOORD 1
  N 1
  ...atoms...
PRIMCOORD 2
  N 1
  ...atoms...
...
```
또는 frame별 `PRIMVEC`로 variable-cell.

**파싱 로직:**
1. `ANIMSTEPS N` 검출 → multi-frame 모드.
2. 각 `PRIMCOORD k` 블록을 한 frame으로. PRIMVEC가 frame마다 있으면
   `latticeMode='per-frame'`, 한 번만 있으면 'fixed'.
3. 각 frame을 `CrystalStructure`로 빌드, `frames[]`에 push.

**통과 기준:**
- 4-frame AXSF fixture: `frames.length === 4`, lattice mode 정확.
- 단발 XSF는 기존 동작 유지 (test-aniso.cif 류 회귀 0).

**작업 단위 추정:** 0.5d.

## 17.1.2 — XDATCAR 파서 (fixed-cell + variable-cell)

**파일:**
- `src/parsers/xdatcarParser.ts` (신규) — `parseXdatcar()` + `parseXdatcarTraj()`.
- `src/parsers/index.ts` — XDATCAR filename association ("XDATCAR" 정확
  match) + `parseStructureFileTraj` dispatch.
- `package.json` — `customEditors.selector`에 `XDATCAR` 추가.
- 신규 fixture: `test/fixtures/test-xdatcar-nve` (10 frames, fixed lattice),
  `test/fixtures/test-xdatcar-npt` (10 frames, variable lattice).

**XDATCAR 형식:**
```
title
1.0
ax ay az
bx by bz
cx cy cz
Ni O
2 2
Direct configuration=     1
0.0 0.0 0.0
...
Direct configuration=     2
0.0 0.0 0.0
...
```

NPT (변수 셀):
```
...
1.0
ax ay az    ← per-frame lattice
...
Ni O
2 2
Direct configuration=     1
...
1.0
ax ay az    ← updated lattice
...
Direct configuration=     2
...
```

**파싱 로직:**
1. 헤더 (title, scale, lattice, species, counts) 한 번 읽음.
2. `Direct configuration=` 마커로 frame 시작 감지.
3. NVE (fixed-cell): 첫 lattice 사용; 모든 frame 같은 species/count.
4. NPT (variable-cell): 각 configuration 앞에 lattice 블록이 있으면 읽고
   업데이트 → `latticeMode='per-frame'`. (VASP NPT XDATCAR는 매 frame 헤더
   반복.)

**통과 기준:**
- NVE fixture: 10 frames, latticeMode='fixed', frames 모두 같은 lattice ref.
- NPT fixture: 10 frames, latticeMode='per-frame', frame별 lattice 변동.
- `parseXdatcar()` (single, 첫 frame) 기존 호출 사이트와 호환.

**작업 단위 추정:** 1d.

## 17.1.3 — Extended XYZ 파서 (ASE format)

**파일:**
- `src/parsers/xyzParser.ts` — 확장. `parseXyzTraj()` 추가.
- 신규 fixture: `test/fixtures/test-extxyz.xyz` (3 frames, ASE-style header).

**Extended XYZ format:**
```
N
Lattice="ax ay az bx by bz cx cy cz" Properties=species:S:1:pos:R:3:forces:R:3 ...
species x y z fx fy fz
species x y z fx fy fz
N
Lattice="..." Properties=...
...
```

**파싱 로직:**
1. 첫 라인 N (atom count).
2. 두 번째 라인 — 이전 XYZ는 free-form comment, ASE는 `Lattice="..." Properties=...`
   key=value 파싱.
3. `Properties` field 파싱: `species:S:1:pos:R:3` → 컬럼 의미 매핑.
   추가 properties (forces, magmom, charge 등)는 v0.17.x에서 처리; 17.1.3
   에서는 species + pos만 필수.
4. N 라인 atom 데이터 → frame 1 완성.
5. EOF 또는 다음 N 라인까지 반복.

**통과 기준:**
- 3-frame extended XYZ fixture 정확 파싱; 매 frame `Lattice` 다르면
  variable-cell.
- 기존 plain XYZ (atom count + comment + atoms) 회귀 0.

**작업 단위 추정:** 1d.

## 17.1.5 — Bond 재계산 토글 + perf 가드

**파일:**
- `src/webview/renderer.ts` — `setFrame()`에서:
  - `recomputeBondsPerFrame === false`: 첫 frame의 `cachedBonds`를 모든 frame
    에 inherit (위치만 update — bondRenderer.rebuild에 새 positions 전달).
  - `true`: 매 frame `detectBonds()` 재호출. 사용자 명시 opt-in이라 slow OK.
- `crystalEditorProvider.ts` — toggle warning hint:
  ```html
  <span class="hint">⚠ slow for >1k atoms</span>
  ```
- 자동 cap: `species.length > 5000` 시 토글 disabled (v0.14의 5k bond skip 패턴).

**통과 기준:**
- Default off + 1k×100 frames 재생 → 30fps.
- Toggle on + 100 atoms × 100 frames → 30fps (작은 N에서는 동작).
- Toggle on + 5k atoms → UI disabled.

**작업 단위 추정:** 0.5d.

## CrystalTrajectory 옵셔널 가드 (전 피처 공통)

전 피처 공통 invariant — 17.1.0 이후 보장:

- `traj.frames.length >= 1` (parser 단계에서 throw if 0 frames).
- `traj.frames[i].species.length` 모든 frame에서 동일 (atom invariance).
- `traj.frames[i].positions.length === traj.frames[i].species.length`.
- `traj.latticeMode === 'fixed'` 시: `traj.frames[i].lattice === traj.frames[0].lattice` (object reference equality, renderer 최적화 의존).
- v0.16 옵셔널 필드 (thermalAniso, occupancy, magMom)는 frame당 별개 (매 frame
  CIF로 export된 trajectory가 frame별 occupancy 가질 수 있음). 첫 컷에서는 frame
  0의 옵셔널 필드만 적용 (변경 시 v0.17.x).

길이 불일치 시 parser throw → editor parse-error boundary가 토스트.
