## 13.1.2 — Editor 에러 바운더리

### 변경 위치

`src/editor/crystalEditorProvider.ts:44-54` (`openCustomDocument`).

### 변경 내용

```typescript
async openCustomDocument(uri, _openContext, _token) {
  const data = await vscode.workspace.fs.readFile(uri);
  const content = new TextDecoder('utf-8').decode(data);
  const filename = path.basename(uri.fsPath);
  try {
    const result = parseStructureFile(content, filename);
    return new CrystalDocument(uri, result.structure, result.volumetric);
  } catch (err: any) {
    const msg = err?.message ?? String(err);
    vscode.window.showErrorMessage(
      `MatViz could not parse ${filename}: ${msg}`,
      'Open as Text'
    ).then(choice => {
      if (choice === 'Open as Text') {
        vscode.commands.executeCommand('vscode.openWith', uri, 'default');
      }
    });
    throw err;  // CustomEditor가 탭을 닫도록 reject 전파
  }
}
```

### Pass criterion

- 일부러 손상된 CIF 파일 (무작위 바이너리) 열기 → 에러 토스트 + "Open as Text" 버튼 표시 → 버튼 클릭 시 텍스트 에디터로 열림.

### `src/parsers/index.ts` 보강 (에러 메시지 품질)

13.1.2의 에러 바운더리가 사용자에게 의미 있는 메시지를 보여줄 수 있도록, QE 파서 호출 전 시그니처 검사를 추가 (fallback이 아니라 **더 나은 에러 메시지 제공 목적**):

```typescript
if (lower.endsWith('.out') || lower.endsWith('.pw') ||
    lower.endsWith('.stdin') || lower.endsWith('.stdout')) {
  if (!looksLikeQEOutput(content)) {
    throw new Error(
      `This .${lower.split('.').pop()} file does not look like Quantum ESPRESSO output ` +
      `(expected CELL_PARAMETERS and/or ATOMIC_POSITIONS). ` +
      `Use "Open as Text" if this was opened by mistake.`
    );
  }
  return { structure: parseQE(content) };
}

function looksLikeQEOutput(content: string): boolean {
  const head = content.slice(0, 4000);
  let hits = 0;
  if (/Program\s+PWSCF/.test(head)) hits++;
  if (content.includes('CELL_PARAMETERS')) hits++;
  if (content.includes('ATOMIC_POSITIONS')) hits++;
  if (content.includes('Begin final coordinates')) hits++;
  return hits >= 2;
}
```

**`.in` / `geometry.in`은 기존 aims 분기가 담당** (`src/parsers/index.ts:41-46`). 거기에도 유사한 `looksLikeAims()` 가드를 추가.

## 13.1.7 — CSP `'unsafe-inline'` 제거

### 변경 위치

`src/editor/crystalEditorProvider.ts:130`.

### 변경 내용

```
// before
style-src ${webview.cspSource} 'unsafe-inline';
// after
style-src ${webview.cspSource};
```

### 사전 검증

`media/styles.css`가 외부 링크로 연결됨 (line 131). HTML 본문에 인라인 `<style>`이나 `style="..."` 속성이 있는지 확인:
- 확인 대상: `getHtml()` template의 html 문자열 전체.
- 알려진 의심 지점: `<canvas id="canvas">`, `<div id="side-panel">` 등 — 현재 읽은 내용에서 inline style은 보이지 않으나 동적 주입 가능성 확인.
- `src/webview/main.ts` 내 `element.style.xxx = ...`는 DOM property assignment이라 CSP `style-src`와 무관.

### Pass criterion

웹뷰 열기 → DevTools Console에 `Content Security Policy: ... style-src ...` 위반 메시지 없음.
모든 스타일(top-bar, side-panel, 버튼, 툴팁) 정상.

### Rollback

위반 발생 시 원복은 한 줄. `'unsafe-inline'` 복구 전에 실제 위반 원인 파악 (inline style 추가된 코드를 외부 CSS로 이동).
